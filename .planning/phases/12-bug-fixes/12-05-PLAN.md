    **BUGF-13: Email verification link includes locale prefix**

    The current email.ts already constructs the URL with locale:
    ```typescript
    const verificationUrl = `${APP_URL}/${locale}/auth/verify-email?token=${token}`;
    ```

    This looks correct IF the locale parameter is being passed. Check the call chain:

    1. **Read `src/features/auth/actions/send-verification-email.ts`** — it passes `locale` to `sendVerificationEmail(userEmail, token, locale)` with a default of `"fr"`.

    2. **Search for all callers of `sendVerificationEmailAction`** to ensure they pass the actual user locale, not just the default:
       ```
       grep -r "sendVerificationEmailAction" src/
       ```
       Check each caller — they should pass the current locale from `useLocale()` or `getLocale()`.

    3. **Common issue: The caller might not pass the locale at all**, relying on the default `"fr"`. Or the APP_URL might already include a trailing slash, causing `//fr/` in the URL.

    4. **Check the APP_URL construction:** `const APP_URL = env.NEXTAUTH_URL ?? "http://localhost:3000";`
       If NEXTAUTH_URL has a trailing slash (e.g., `http://localhost:3000/`), the URL would be `http://localhost:3000//fr/auth/verify-email`. Fix by stripping trailing slashes:
       ```typescript
       const APP_URL = (env.NEXTAUTH_URL ?? "http://localhost:3000").replace(/\/+$/, "");
       ```

    5. **Also check the password reset URL** in the same file — apply the same trailing slash fix.

    6. **Verify the verify-email page route exists** at `src/app/[locale]/(client)/auth/verify-email/page.tsx`. Read it to ensure it handles the `token` search param correctly.

    7. **If callers are not passing locale**: Update the callers to pass the actual locale. Search for patterns like:
       - In registration flow: `sendVerificationEmailAction(userId, email)` should be `sendVerificationEmailAction(userId, email, locale)`
       - In resend verification: same pattern

    The fix should be minimal — just ensure locale is always passed and APP_URL doesn't double-slash.

    **BUGF-14: Provider profile zone selector allows selecting intervention zones**

    Read the ZoneSelector component carefully. It currently:
    - Uses `useState<Set<string>>` for selectedIds (initialized from `initialDelegationIds`)
    - Has toggle/selectAll/deselectAll/remove functions
    - Calls `updateZonesAction` on save

    Potential issues to investigate and fix:

    1. **Set state immutability:** The current implementation creates new Set instances in each state update, which looks correct. But verify React is re-rendering properly — Sets can sometimes cause issues with React's shallow comparison. If toggleDelegation doesn't trigger a re-render, the checkbox won't visually update.

    2. **Read `src/features/provider/actions/manage-zones.ts`** to check the server action:
       - Does it delete existing zones and replace with new ones? (correct approach)
       - Or does it try to diff? (could cause bugs)
       - It should: delete all existing ProviderDelegation records for this provider, then create new ones for the passed delegationIds

    3. **Check the Checkbox component interaction:** The shadcn Checkbox uses `onCheckedChange` callback. The current code uses `onCheckedChange={() => toggleDelegation(del.id)}` which should work. But if the Checkbox expects a boolean parameter, the toggle might not fire correctly. Test: does `onCheckedChange` pass the new checked state? If so, consider using it:
       ```tsx
       onCheckedChange={(checked) => {
         setSelectedIds((prev) => {
           const next = new Set(prev);
           if (checked) {
             next.add(del.id);
           } else {
             next.delete(del.id);
           }
           return next;
         });
       }}
       ```

    4. **Check if gouvernorats data is being loaded properly.** Read `src/app/[locale]/(provider)/provider/profile/edit/page.tsx` to see how gouvernorats are fetched and passed to ZoneSelector. If the data isn't loaded, the selector shows empty.

    5. **Check the updateZonesAction implementation** in manage-zones.ts:
       - Ensure it uses a transaction to delete old zones and create new ones atomically
       - Ensure it handles the case where delegationIds is empty (should it allow saving 0 zones? The component blocks this with `disabled={selectedIds.size === 0}`)

    6. **If the action is failing silently**, add better error handling/feedback. Check if the action returns proper error messages that the toast can display.

    Apply minimal fixes — the component logic looks sound. The bug is likely in the server action or in how the initial data is loaded. Fix whatever is actually broken.
