**Cancellation utility (`src/lib/utils/cancellation.ts`):**
Create a pure function with zero side effects:

```typescript
export type CancellationTier = "FULL" | "PARTIAL" | "NONE";

export interface CancellationResult {
  tier: CancellationTier;
  refundPercentage: number; // 0, 50, or 100
  hoursUntilScheduled: number;
}

export function calculateRefundPercentage(scheduledAt: Date, now?: Date): CancellationResult
```

Logic (per 06-CONTEXT.md):
- If scheduledAt is more than 48 hours from now: tier="FULL", refundPercentage=100
- If scheduledAt is 24-48 hours from now: tier="PARTIAL", refundPercentage=50
- If scheduledAt is less than 24 hours from now: tier="NONE", refundPercentage=0
- Calculate hoursUntilScheduled as (scheduledAt - now) / (1000 * 60 * 60)
- If scheduledAt is in the past: tier="NONE", refundPercentage=0

Accept optional `now` parameter for testability (defaults to new Date()).

**Cancel booking action (`src/features/booking/actions/cancel-booking.ts`):**

`cancelBookingAction(data)`:
- Validates CLIENT role session
- Parses with cancelBookingSchema (from booking.ts validations — import from Plan 01)
- Verifies booking belongs to client (clientId = user.id)
- Verifies booking status is PENDING or ACCEPTED (cannot cancel IN_PROGRESS or COMPLETED)
- If status is PENDING or ACCEPTED:
  - Calculate refund using calculateRefundPercentage(booking.scheduledAt)
  - In a $transaction:
    - Update booking: status = CANCELLED, cancelledAt = now(), cancelledBy = "CLIENT", cancelReason = data.reason
    - If payment exists and refundPercentage > 0: update payment refundAmount = amount * refundPercentage / 100, status = REFUNDED if 100%, refundedAt = now()
    - If refundPercentage === 0: payment stays as PENDING (no refund)
  - Return `{ success: true, refund: { tier, refundPercentage, refundAmount } }`
- If booking has no scheduledAt (edge case for quote-based bookings not yet scheduled): default to FULL refund

Also add `cancelBookingProviderAction(bookingId, reason?)`:
- Validates PROVIDER role session
- Verifies booking belongs to provider, status is PENDING or ACCEPTED
- Updates status CANCELLED, cancelledBy = "PROVIDER", always FULL refund (provider-initiated = 100% refund to client)
- Returns `{ success: true }`

**Quote expiration logic (`src/features/booking/actions/expire-quotes.ts`):**

1. `checkAndExpireQuote(quoteId: string)`:
   - Finds quote by id
   - If status is PENDING and expiresAt < now(): update status to EXPIRED
   - Returns boolean (true if expired)
   - Used inline before quote operations (lazy expiration check)

2. `expireQuotesAction()`:
   - Batch operation: find all quotes where status = PENDING AND expiresAt < now()
   - Update all matching quotes: status = EXPIRED
   - Returns `{ expired: count }`
   - No session check needed — this is called by the cron endpoint

**Cron endpoint (`src/app/api/cron/expire-quotes/route.ts`):**

Create a GET handler:
- Check for Authorization header matching `Bearer ${process.env.CRON_SECRET}` (standard Vercel Cron auth pattern)
- If CRON_SECRET is not set in env (development), allow the request (log warning)
- Call expireQuotesAction()
- Return JSON `{ expired: N, timestamp: ISO string }`

Add CRON_SECRET to src/env.ts as optional string (not required — dev mode works without it).

Add to vercel.json (create if not exists):
```json
{
  "crons": [{
    "path": "/api/cron/expire-quotes",
    "schedule": "0 */6 * * *"
  }]
}
```

This runs every 6 hours. Combined with lazy check in quote operations, ensures timely expiration.

**Important:** The lazy check (checkAndExpireQuote) is the primary mechanism — it runs before any quote read/mutation in Plan 01's quote actions. The cron is a secondary sweep for quotes that nobody queries.
